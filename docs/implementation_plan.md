# 淘汰赛全图交互优化方案

## 问题描述
用户反馈在“查看全图”模式下：
1. 无法滑动页面（Panning 不响应）。
2. 需要实现双指捏合（Pinch-to-zoom）缩放功能。

## 根本原因分析
1. **滑动受限**：当前的 `handleMouseDown` 逻辑中过滤了按钮（PlayerCard），导致用户在选手卡片上点击并拖拽时无法触发滑动。由于卡片占据了大部分可视区域，导致用户感到“无法滑动”。
2. **缩放逻辑**：虽然已有缩放代码，但可能由于事件冒泡、`touch-action` 或计算逻辑在某些设备上不够平滑，导致用户体验不佳。
3. **初始视图**：目前初始缩放比例固定为 1，对于大型对阵图，可能导致初始只能看到局部且不知道可以滑动。

## 拟改进内容

### 1. 优化交互逻辑 (`BracketModal`)
- **允许从卡片开始滑动**：移除 `handleMouseDown` 中对按钮的限制，改由 `suppressClickRef` 区分点击与拖拽。
- **完善双指缩放**：
    - 优化 `handleTouchMove` 中的多点触控计算。
    - 确保在缩放过程中保持缩放中心点（Focal Point）稳定。
- **防止浏览器干扰**：确保在所有滑动/缩放操作中调用 `e.preventDefault()`。

### 2. 初始视图自动适配
- **Fit to Screen**：在 Modal 打开时，根据对阵图的总宽高和窗口宽高，计算一个合适的初始缩放比例（Scale）和位置（Position），确保用户第一眼能看到全貌。

### 3. 代码实现细节 ([page.tsx](file:///e:/work/fencing-championship-v2/app/%5BsportCode%5D/event/%5Bname%5D/brackets/page.tsx))
- 修改 `handleMouseDown` 逻辑。
- 优化 `handleTouchStart/Move/End` 逻辑。
- 在 `useEffect` 的 `isOpen` 分支中增加视图适配计算。

## 验证方案
### 手动验证
- **电脑端**：使用鼠标拖拽全图各处（包括选手卡片），确认可以自由滑动；使用滚轮确认缩放正常。
- **移动端**：
    - 单指拖拽全图（包括卡片），确认滑动顺畅。
    - 双指捏合/张开，确认缩放平滑且以双指中心为缩放点。
    - 短促点击选手卡片，确认仍能正确触发跳转。
